# 🔍 NAILED-IT 코드 리뷰

> 리뷰 대상 브랜치: `develop`
> 리뷰 범위: `index.html`, `login.html`, `signup.html`, `admin-login.html`, `categories.html`, `src/main.js`, `src/style.css`
> 리뷰 관점: **응집도는 높이고, 결합도는 낮추자**

먼저 결론부터 말씀드리면, 지금 코드는 **작동은 잘 하지만 "같은 코드가 여러 곳에 반복"되는 상태**예요. 지금은 화면이 5개라 괜찮아 보여도, 화면이 10개, 20개로 늘어나면 같은 수정을 여러 파일에 반복해서 해야 하는 순간이 옵니다. 이 리뷰는 그런 문제가 생기기 전에 구조를 정돈하는 데 초점을 맞췄습니다.

리뷰는 **칭찬 → 문제점 → 왜 문제인지** 순서로 설명할게요. 해결 방법은 별도 문서 [`refactoring-guide.md`](./refactoring-guide.md) 에 정리해 두었습니다.

## 📖 용어부터 쉽게

리뷰에서 자주 쓸 두 단어를 먼저 설명할게요.

- **응집도(Cohesion)**: "한 파일(또는 한 함수) 안의 코드들이 서로 얼마나 관련 있는가?"
  - 예: `login.js` 파일 안에 로그인 관련 코드만 있으면 응집도가 **높음** 👍
  - 예: `utils.js` 안에 로그인, 회원가입, 카테고리, 날짜 포맷, 색깔 코드… 다 섞여 있으면 응집도가 **낮음** 👎
  - **응집도가 높다 = 한 곳만 보면 된다**
- **결합도(Coupling)**: "A 파일을 고쳤을 때 B 파일도 같이 고쳐야 하는가?"
  - 예: HTML 안에 `onclick="alert(...)"` 가 박혀 있음 → HTML과 JS가 딱 붙어 있어서 결합도 **높음** 👎
  - 예: HTML에는 `id="loginBtn"` 만 있고, JS에서 이벤트를 연결 → 결합도 **낮음** 👍
  - **결합도가 낮다 = 한 파일 수정이 다른 파일에 영향을 덜 준다**

우리 목표는 **관련 있는 건 한 곳에 모으고(응집도 ↑), 서로 떨어져 있어야 하는 건 분리(결합도 ↓)** 하는 거예요.

## ✨ 잘한 점 (먼저 칭찬부터!)

이건 진짜로 잘 한 부분이에요. 빈말 아니에요.

1. **Tailwind CSS 클래스를 적극적으로 잘 사용했습니다.**
   별도 CSS 파일에 스타일을 몰아 쓰지 않고, 화면 단위로 Tailwind 유틸리티 클래스를 사용한 점이 좋아요. `src/style.css` 도 `@import "tailwindcss";` 한 줄만 있어서 깔끔해요.

2. **디자인 톤이 일관됩니다.**
   모든 페이지가 `max-w-md mx-auto` 로 모바일 폭을 맞추고, `border-black/10`, `rounded-lg`, `py-4` 같은 값을 반복 사용해서 시각적으로 통일감이 있어요.

3. **HTML 주석으로 섹션을 잘 구분했습니다.**
   `<!-- Header -->`, `<!-- Main Content -->` 처럼 주석으로 구역을 나눠 놓아서 코드를 처음 읽는 사람도 구조를 파악하기 쉬워요.

4. **폼 요소에 `id`가 붙어 있어요.**
   `#loginForm`, `#email`, `#password` 같은 id가 이미 잡혀 있어서 나중에 JavaScript에서 붙잡기 쉽게 준비가 되어 있습니다.

좋은 기반 위에 올라가 있어요. 여기서부터가 진짜 리뷰입니다.

## 1️⃣ 헤더가 파일마다 복붙되어 있어요 (응집도 ↓, 결합도 ↑)

`login.html`, `signup.html`, `admin-login.html`, `categories.html` 을 열어 보면 **뒤로가기 화살표 + 제목** 구조의 헤더가 거의 똑같이 반복됩니다.

예를 들어 `login.html:13-25` 와 `signup.html:13-25` 는 제목 글자("로그인" vs "회원가입")만 빼면 완전히 같아요.

```html
<!-- login.html -->
<header class="px-6 py-4 border-b border-black/10">
  <div class="max-w-md mx-auto flex items-center gap-4">
    <a href="index.html" class="p-2 -ml-2 hover:bg-black/5 rounded-lg transition-colors">
      <svg ...>...</svg>
    </a>
    <h2 class="font-medium">로그인</h2>
  </div>
</header>
```

### 왜 문제일까?

- **수정할 때 네 군데를 다 고쳐야 해요.**
  예를 들어 디자이너가 "헤더 아래쪽 테두리 색을 진하게 해주세요" 라고 하면, 네 개의 파일을 열어서 `border-black/10` 을 `border-black/20` 으로 각각 고쳐야 합니다. **하나라도 빼먹으면 그 페이지만 디자인이 달라지는 버그**가 생겨요. 이게 바로 "결합도가 높다"는 상태예요.
- **읽는 사람도 힘들어요.**
  새로 합류한 팀원이 "아, 이 헤더가 다 똑같은 거구나" 라고 스스로 알아차려야 합니다. 코드가 말을 해 주지 않아요.

## 2️⃣ 같은 SVG 아이콘이 파일마다 복붙되어 있어요

SVG 아이콘(뒤로가기 화살표, 홈, 히스토리, 로그아웃, HTML/CSS/JS 로고)이 **한 파일 안에서도 여러 번**, **여러 파일에 걸쳐서도 여러 번** 반복됩니다.

예를 들어 HTML 카테고리 아이콘을 찾아 보세요.

- `index.html:40-45` 에 한 번
- `categories.html:79-85` 에 또 한 번

```html
<svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none"
     stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round">
  <polyline points="16 18 22 12 16 6"></polyline>
  <polyline points="8 6 2 12 8 18"></polyline>
  <line x1="12" y1="2" x2="12" y2="22"></line>
</svg>
```

똑같은 SVG 코드 블록이 프로젝트 전체에 **9회 이상** 반복됩니다.

### 왜 문제일까?

- 아이콘 모양을 바꾸려면 9곳을 다 찾아서 고쳐야 해요.
- HTML 파일이 불필요하게 길어져서, 진짜 중요한 **페이지 콘텐츠**가 뭔지 읽기 어려워집니다. 지금 `categories.html` 은 **159줄** 인데, 이 중 아이콘 SVG가 차지하는 줄이 절반 가까이 돼요. 본문보다 장식이 더 긴 느낌이에요.

---

## 3️⃣ 로그인 폼과 관리자 로그인 폼이 거의 똑같습니다

`login.html:30-77` 와 `admin-login.html:41-78` 을 비교해 보세요.

차이라고는 **폼의 `id` (`loginForm` vs `adminLoginForm`), 위쪽의 방패 아이콘, "비밀번호 찾기/회원가입" 링크 유무**. 이게 전부예요. 이메일 입력칸, 비밀번호 입력칸, "로그인 상태 유지" 체크박스, 에러 메시지, 제출 버튼은 **문자 그대로 같은 코드**가 두 벌 존재합니다.

### 왜 문제일까?

- 예를 들어 "비밀번호 입력칸 아래에 '비밀번호를 잊으셨나요?' 라는 힌트를 추가해달라" 는 요청이 오면, 두 파일을 똑같이 고쳐야 해요.
- 심지어 "한쪽은 고쳤는데 다른 쪽은 잊어버리고 안 고친" 상태가 되면, 같은 브랜드의 페이지인데 화면이 서로 달라 보여서 **사용자가 혼란을 느낍니다.**

## 4️⃣ 카테고리 버튼 3개가 "거의 똑같은 모양"으로 3번 반복됩니다

`categories.html:74-143` 은 HTML / CSS / JavaScript 카테고리 버튼 세 개를 **거의 똑같은 마크업**으로 연달아 작성하고 있어요. 바뀌는 건 **아이콘 SVG**, **제목("HTML" / "CSS" / "JavaScript")**, **설명 문구** 세 가지뿐인데, 버튼 전체 70줄이 반복됩니다.

`index.html:36-71` 에서도 같은 3개 카테고리 아이콘 카드가 또 반복되고요.

### 왜 문제일까?

- 카테고리를 하나 추가한다고 생각해 봅시다. 예를 들어 "React" 카테고리가 생기면, **똑같이 생긴 버튼 코드를 네 번째로 복사해서 붙여 넣어야** 해요. 이게 3개, 4개, 5개, … 늘어날수록 파일이 폭발합니다.
- 나중에 "카테고리에 문제 수(10문제)" 대신 "진행률 바" 를 넣어 달라는 요청이 오면, **똑같은 수정을 3번 반복**해야 합니다.

## 5️⃣ `onclick="alert(...)"` 인라인 핸들러를 쓰고 있어요

`login.html:67` 에 이런 코드가 있습니다.

```html
<button type="button" onclick="alert('비밀번호 찾기 기능은 추후 구현 예정입니다.')" ...>
  비밀번호 찾기
</button>
```

편리하지만, 이건 **HTML 속에 JavaScript가 박혀 있는 상태**예요. HTML(구조)과 JavaScript(동작)가 한 줄에서 뒤섞여 있습니다.

### 왜 문제일까?

- **HTML과 JS가 결합도 높게 붙어 있어요.** 지금은 `alert` 한 줄이라 괜찮지만, 나중에 이 버튼이 "모달을 띄우고, 이메일을 입력받고, 서버에 API를 쏜다" 로 기능이 커지면 `onclick` 속성 하나에 그 모든 로직이 다 들어갈 수 없습니다. 결국 JS로 옮기게 되는데, 그때 가서 옮기면 어차피 **이중 작업**이에요.
- **CSP(Content Security Policy)** 같은 보안 정책이 적용되면 인라인 핸들러가 차단될 수 있어요.
- 이벤트 리스너를 여러 개 붙이기가 어렵습니다.

## 6️⃣ `<label>` 에 `for` 속성이 없어요 (접근성)

`login.html:33-38` 예시:

```html
<div class="space-y-2">
  <label class="block text-sm">이메일</label>
  <input type="email" id="email" ... />
</div>
```

### 왜 문제일까?

- **스크린 리더(시각장애인을 위한 화면 읽기 프로그램)가 "이메일" 이라는 라벨과 입력칸을 연결하지 못합니다.** 사용자는 "편집 가능한 텍스트 필드" 라는 안내만 듣고, 여기가 이메일 칸인지 비밀번호 칸인지 알 수 없어요.
- **라벨 글자를 클릭해도 입력칸으로 포커스가 이동하지 않습니다.** 원래는 `<label for="email">` 로 연결하면 라벨 글자만 클릭해도 입력칸이 활성화돼요. 이건 모바일에서 특히 유용합니다.

이건 사용자를 **실제로 불편하게 만드는** 문제라서, 심각도가 낮아 보여도 꼭 고쳐야 합니다.

---

## 7️⃣ 아이콘 전용 버튼/링크에 `aria-label` 이 없어요

`categories.html:28-52` 에는 홈 / 히스토리 / 로그아웃 아이콘 버튼이 있는데, 버튼 안에 글자가 없고 SVG만 있어요.

```html
<button id="logoutBtn" class="p-2 hover:bg-black/5 rounded-lg transition-colors">
  <svg ...>...</svg>
</button>
```

### 왜 문제일까?

- 스크린 리더로 들으면 "버튼" 이라고만 읽힙니다. 이게 로그아웃 버튼인지 홈 버튼인지 알 수 없어요.
- 해결은 쉽습니다. `aria-label="로그아웃"` 한 줄만 추가하면 돼요.

---

## 8️⃣ `<input>` 에 `name`, `required` 속성이 빠져 있어요

`login.html`, `signup.html`, `admin-login.html` 의 모든 입력 필드에서:

- `name` 이 없어서 → 나중에 JS 없이 순수 HTML 폼 전송을 하려고 해도 서버가 값을 읽지 못합니다.
- `required` 가 없어서 → 빈 값으로 제출 버튼을 눌러도 브라우저가 막아주지 않아요. 이걸 JS로 막으려면 **직접 검증 코드**를 짜야 합니다. HTML이 공짜로 해 주는 일을 JS로 다시 짜는 건 손해예요.
- `autocomplete` 도 없어서, 브라우저가 저장된 이메일/비밀번호를 자동으로 채워주지 못합니다. `autocomplete="email"`, `autocomplete="current-password"`, `autocomplete="new-password"` 를 써주면 사용자 경험이 크게 좋아져요.

---

## 9️⃣ 카테고리 버튼이 어디로도 이동하지 않습니다 (기능 미완성)

`categories.html:75-143` 의 세 카테고리 버튼은 `<button>` 으로 만들어져 있는데, **클릭했을 때 어디로 이동할지 아무런 정보가 없습니다.** `onclick` 도 없고, `href` 도 없고, `data-*` 속성도 없어요.

### 왜 문제일까?

- 지금 상태에서는 이 페이지의 핵심 기능이 **동작하지 않습니다**. 사용자가 카테고리를 고를 방법이 없어요.
- "페이지 이동"이 목적이면 `<button>` 이 아니라 `<a href="quiz.html?category=html">` 가 더 적절해요. 버튼은 "액션(form submit 등)" 에 쓰고, 링크는 "이동" 에 써야 HTML 시맨틱이 맞습니다.

## 🔟 `<h1>` 이 `index.html` 안에 두 개 있어요

`index.html:16` 과 `index.html:26` 에 각각 `<h1>` 이 있어요. 하나는 헤더의 작은 로고("NAILED-IT"), 다른 하나는 본문의 큰 타이틀("NAILED-IT") 입니다.

### 왜 문제일까?

- HTML 시맨틱 규칙상 한 페이지에 `<h1>` 은 **하나**가 권장돼요. 검색 엔진이나 접근성 도구는 `<h1>` 을 "이 페이지가 무엇에 관한 것인지"를 알려주는 간판으로 읽습니다. 간판이 두 개면 혼란스러워요.
- 해결은 "헤더의 작은 로고는 `<h1>` 이 아니라 `<span>` 또는 `<a>` 로 바꾸기" 혹은 "본문 타이틀을 `<h2>` 로 바꾸기" 입니다.

## 1️⃣1️⃣ README 구조와 실제 파일 구조가 다릅니다

`README.md:38-59` 는 이렇게 말하고 있어요.

```
NAILED-IT/
├── index.html
├── src/
│ ├── pages/
│ │ ├── login.html
│ │ ├── signup.html
│ │ ├── category.html
│ │ ...
```

그런데 실제로는 `login.html`, `signup.html`, `admin-login.html`, `categories.html` 이 **프로젝트 루트**에 있습니다. `src/` 안에는 `main.js` 와 `style.css` 만 있어요.

### 왜 문제일까?

- 새로 합류한 사람이 README만 보고 `src/pages/login.html` 을 찾으면 당연히 없습니다. "뭐지? 내가 뭔가 잘못했나?" 하고 혼란스러워해요.
- 문서와 실제 구조가 맞지 않는 건 **말과 행동이 다른 상태**예요. 둘 중 하나를 맞춰야 합니다.

---

## 1️⃣2️⃣ `main.js` 가 거의 비어 있어요

`src/main.js` 는 현재 이 한 줄입니다.

```js
import "./style.css";
```

그리고 모든 HTML 파일이 `<script type="module" src="/src/main.js"></script>` 를 똑같이 불러와요.

### 왜 문제일까?

- 지금은 **페이지별 JavaScript 로직을 작성할 자리가 없습니다.** 로그인 페이지는 로그인 폼 submit 을 처리해야 하고, 카테고리 페이지는 카테고리 버튼 클릭을 처리해야 하는데, 그걸 어디에 둘지가 정해져 있지 않아요.
- 모든 HTML이 **같은 `main.js`** 를 부르고 있어서, 나중에 페이지별 로직을 추가하면 **로그인 페이지에서 카테고리 JS도 같이 실행되는** 이상한 상황이 생기기 쉬워요.
