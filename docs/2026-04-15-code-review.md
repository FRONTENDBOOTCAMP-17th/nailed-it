## 먼저 칭찬할 점

- 화면의 기본 레이아웃이 깔끔하고, Tailwind CSS로 빠르게 UI를 만들려는 방향은 좋습니다.
- 로그인, 회원가입, 카테고리 선택처럼 사용자 흐름을 먼저 화면으로 잡아둔 점도 좋습니다.
- `src/style.css`에 `@import "tailwindcss";`만 두고 시작한 점은 "불필요한 CSS를 줄이자"는 방향과 맞습니다.

하지만 지금 상태는 "화면 시안"에 가깝고, 실제 서비스 흐름으로 보면 중요한 문제가 몇 가지 있습니다. 아래 항목부터 우선적으로 고치는 것을 권장합니다.

## 중요도 높음

### 1. 배포 결과물에 `index.html`만 포함됩니다

**위치**

- `vite.config.js:1-6`
- `index.html`
- `login.html`
- `signup.html`
- `categories.html`
- `admin-login.html`

**문제 설명**

- 현재 `vite build` 결과물에는 `dist/index.html`만 생성됩니다.
- 즉, 로그인/회원가입/카테고리/관리자 로그인 페이지를 열 수 있는 개발용 HTML 파일은 있어도, 실제 배포 결과물에는 포함되지 않습니다.
- 사용자는 홈 화면까지만 보고, 그 다음 화면으로는 정상 이동할 수 없습니다.

**왜 문제인가요?**

- 초보 개발자 입장에서는 "HTML 파일이 있으니 배포도 되겠지"라고 생각하기 쉽습니다.
- 하지만 Vite는 지금 설정대로 두면 여러 HTML 파일을 자동으로 모두 배포해주지 않습니다.
- 그래서 화면을 여러 개 만들었어도 실제 서비스에서는 일부만 보이게 됩니다.

**리뷰 의견**

- 이 프로젝트는 여러 페이지가 필요하므로, "멀티 페이지 앱(MPA)"로 빌드 설정을 명확히 하거나, 아예 하나의 HTML 안에서 화면을 바꾸는 방식으로 구조를 정해야 합니다.
- 현재 요구사항과 파일 구성을 보면, 우선은 MPA 구조로 정리하는 것이 초보자에게 더 이해하기 쉽습니다.

### 2. 핵심 기능이 아직 연결되지 않았습니다

**위치**

- `src/main.js:1`
- `login.html:30-77`
- `signup.html:30-81`
- `admin-login.html:41-78`
- `categories.html:71-145`

**문제 설명**

- `src/main.js`에는 지금 CSS import만 있고, 로그인/회원가입/로그아웃/카테고리 선택에 대한 JavaScript 로직이 없습니다.
- 각 폼에는 제출 버튼이 있지만 `submit` 이벤트 처리 코드가 없습니다.
- 카테고리 버튼도 클릭 시 어떤 페이지로 이동할지, 어떤 API를 부를지 연결이 없습니다.

**왜 문제인가요?**

- 지금 화면은 "보이기만 하는 상태"입니다.
- 사용자는 버튼을 눌러도 로그인되지 않고, 회원가입도 되지 않으며, 카테고리를 골라도 퀴즈를 시작할 수 없습니다.
- `NAILED_IT_API_REQUIREMENTS.md`에는 인증/카테고리/세션 API 흐름이 자세히 적혀 있는데, 현재 구현은 그 흐름과 거의 연결되지 않았습니다.

**리뷰 의견**

- UI 작업과 기능 작업을 분리해서 생각해야 합니다.
- "HTML 만들기"와 "동작 연결하기"는 다른 일입니다.
- 지금은 HTML이 먼저 만들어졌으니, 다음 단계에서는 각 페이지별 JavaScript 파일을 만들고 페이지 진입 시 필요한 동작을 연결해야 합니다.

### 3. 존재하지 않는 페이지로 이동하는 링크가 있습니다

**위치**

- `categories.html:36`
- `README.md:40-56`

**문제 설명**

- `categories.html`에는 `history.html`로 가는 링크가 있습니다.
- 하지만 현재 저장소에는 `history.html` 파일이 없습니다.
- `README.md`에는 `quiz.html`, `result.html`, `history.html`, 관리자 페이지들이 있는 것처럼 적혀 있지만 실제 파일은 없습니다.

**왜 문제인가요?**

- 사용자는 클릭 후 404 페이지를 보게 됩니다.
- 문서와 실제 코드가 다르면 팀원끼리 오해가 생깁니다.
- 특히 초보 팀에서는 "문서에 있으니 구현돼 있겠지"라고 착각하기 쉽습니다.

**리뷰 의견**

- 문서와 실제 코드는 항상 같이 움직여야 합니다.
- 아직 만들지 않은 페이지라면 링크를 잠시 제거하거나 비활성화해야 합니다.
- 이미 설계만 끝난 상태라면 README에는 "예정 구조"라고 명확히 적는 편이 좋습니다.

## 중요도 중간

### 4. 같은 구조가 여러 파일에 반복되어 결합도가 높아졌습니다

**위치**

- `index.html:13-91`
- `login.html:13-75`
- `signup.html:13-79`
- `categories.html:13-150`
- `admin-login.html:13-74`

**문제 설명**

- 헤더 구조, 입력창 스타일, 버튼 스타일, 카드 스타일이 여러 HTML 파일에 거의 같은 형태로 반복됩니다.
- 예를 들어 `max-w-md mx-auto`, `border border-black/10 rounded-lg`, `w-full bg-black text-white py-4 rounded-lg` 같은 묶음이 계속 반복됩니다.

**왜 문제인가요?**

- 한 곳의 디자인을 바꾸면 모든 파일을 직접 수정해야 합니다.
- 예를 들어 버튼 모서리 값이나 여백을 바꾸고 싶으면 로그인, 회원가입, 관리자 로그인 등 여러 파일을 다 찾아가며 수정해야 합니다.
- 이런 구조는 결합도가 높고, 실수도 많이 생깁니다.

**리뷰 의견**

- "공통 UI 조각"을 정리하는 습관이 필요합니다.
- 다만 여기서 CSS 파일을 크게 늘리라는 뜻은 아닙니다.
- Tailwind를 계속 쓰되, JavaScript 함수로 공통 마크업을 만들거나, 데이터 배열을 순회해서 반복 UI를 렌더링하는 방식이 더 적절합니다.

---

### 5. 폼 접근성과 기본 HTML 검증이 부족합니다

**위치**

- `login.html:34-45`
- `signup.html:34-60`
- `admin-login.html:45-56`

**문제 설명**

- 여러 `label`에 `for` 속성이 없습니다.
- 입력창에 `name`, `required`, `autocomplete` 같은 기본 속성이 없습니다.
- 회원가입 화면은 안내 문구는 있지만 브라우저 기본 검증을 잘 활용하지 못하고 있습니다.

**왜 문제인가요?**

- 마우스 없이 키보드만 사용하는 사용자에게 불편합니다.
- 브라우저가 기본적으로 도와줄 수 있는 검증 기능을 놓치고 있습니다.
- JavaScript로 모든 검증을 직접 처리해야 해서 코드가 더 길어지고 복잡해집니다.

**리뷰 의견**

- HTML 자체가 가진 기능을 먼저 활용한 뒤, 부족한 부분만 JavaScript로 보완하는 습관이 좋습니다.
- 초보일수록 "브라우저가 기본으로 해주는 것"을 적극 활용해야 실수를 줄일 수 있습니다.

---

### 6. 인라인 이벤트는 구조를 흐리게 만듭니다

**위치**

- `login.html:67`

**문제 설명**

- `onclick="alert(...)"` 형태의 인라인 이벤트가 들어가 있습니다.
- 이 방식은 HTML 안에 동작 코드가 섞여 들어가게 만듭니다.

**왜 문제인가요?**

- 구조(HTML)와 동작(JavaScript)이 섞이면 파일 역할이 흐려집니다.
- 나중에 기능이 늘어나면 HTML 파일이 금방 복잡해집니다.

**리뷰 의견**

- 이벤트 연결은 JavaScript에서 `addEventListener`로 처리하는 습관을 들이는 편이 좋습니다.
- 이렇게 해야 파일 역할이 분리되고, 응집도도 더 높아집니다.

## 우선순위 제안

1. 여러 페이지가 실제로 배포되도록 Vite 구조를 먼저 정리합니다.
2. 로그인, 회원가입, 관리자 로그인, 카테고리 선택에 페이지별 JavaScript를 연결합니다.
3. 없는 페이지 링크를 제거하거나 실제 파일을 만듭니다.
4. 반복되는 UI 구조를 공통 함수/데이터 기반 렌더링으로 줄입니다.
5. `label`, `required`, `name`, `autocomplete` 등 기본 HTML 품질을 올립니다.

## 한 줄 총평

지금 코드는 "디자인 시안 단계"로는 출발이 좋지만, 서비스 코드로 보기에는 기능 연결과 구조 분리가 아직 부족합니다. 특히 "화면은 있는데 동작하지 않는 상태"와 "문서에는 있는데 실제 파일이 없는 상태"를 먼저 정리해야 다음 단계 개발이 훨씬 쉬워집니다.
