## 1. 가장 먼저 해야 할 일: 이름과 경로를 한 번에 맞추기

지금 가장 큰 문제는 "파일명은 바뀌었는데, 그 파일을 참조하는 곳들이 같이 안 바뀐 상태"입니다.

이 문제는 아래 네 군데를 같이 확인해야 해결됩니다.

1. 실제 파일명
2. HTML 안의 `href`
3. `vite.config.js`의 `input`
4. `README.md`의 구조 설명

### 꼭 기억할 규칙

- 파일명을 바꾸면 링크도 같이 바꿉니다.
- 파일명을 바꾸면 빌드 설정도 같이 바꿉니다.
- 파일명을 바꾸면 문서도 같이 바꿉니다.

이 세 가지를 한 묶음으로 생각해야 합니다.

## 2. 현재 프로젝트에서 먼저 통일해야 할 파일명

지금 실제 파일 기준으로 맞추려면 아래처럼 생각하면 됩니다.

```text

pages/result.html

pages/admin/quiz-list.html
pages/admin/quiz-form.html

pages/admin/quiz-detail.html
```

그러면 `vite.config.js`도 이 이름을 기준으로 바뀌어야 합니다.

예시:

```js
input: {
  main: resolve(__dirname, "index.html"),
  login: resolve(__dirname, "pages/login.html"),
  signup: resolve(__dirname, "pages/signup.html"),
  category: resolve(__dirname, "pages/category.html"),
  quiz: resolve(__dirname, "pages/quiz.html"),
  result: resolve(__dirname, "pages/result.html"),

  adminQuizDetail: resolve(__dirname, "pages/admin/quiz-detail.html"),
```

### 왜 이것이 중요한가요?

- 이 단계가 맞아야 `npm run build`가 됩니다.
- 빌드가 되어야 배포도 되고, 팀원이 결과물을 확인할 수도 있습니다.

## 3. 상대 경로를 정확히 이해해야 합니다

초보 개발자가 가장 많이 헷갈리는 부분이 경로입니다.

예를 들어 `pages/login.html` 안에서 아래처럼 쓰면:

```html
<a href="index.html">뒤로가기</a>
```

브라우저는 `pages/index.html`을 찾습니다. 하지만 실제 홈 화면은 루트의 `index.html`입니다.

그래서 올바른 경로는 이렇게 써야 합니다.

```html
<a href="../index.html">뒤로가기</a>
```

### 간단한 기준

- 현재 파일이 루트에 있으면: `pages/login.html`
- 현재 파일이 `pages/` 안에 있으면: `../index.html`
- 현재 파일이 `pages/admin/` 안에 있으면:
  - 같은 폴더 파일로 이동: `quiz-list.html`
  - `pages/` 폴더 파일로 이동: `../category.html`
  - 루트 파일로 이동: `../../index.html`

### 실수 줄이는 팁

- 링크를 고친 뒤 꼭 직접 눌러봅니다.
- 페이지를 옮긴 날에는 링크 검사도 같이 합니다.

## 4. `main.js`는 "페이지를 시작시키는 파일"로 써야 합니다

지금 `main.js`는 CSS만 import 하고 있습니다.

앞으로는 아래 역할만 맡기세요.

- 공통 CSS 로드
- 현재 페이지 확인
- 해당 페이지 초기화 함수 실행

예시:

```js
import "./style.css";
import { initLoginPage } from "./pages/login.js";
import { initSignupPage } from "./pages/signup.js";
import { initCategoryPage } from "./pages/category.js";
import { initHistoryPage } from "./pages/history.js";
import { initQuizPage } from "./pages/quiz.js";
import { initResultPage } from "./pages/result.js";
import { initQuizListPage } from "./pages/admin/quiz-list.js";
import { initQuizFormPage } from "./pages/admin/quiz-form.js";
import { initQuizEditPage } from "./pages/admin/quiz-edit.js";
import { initQuizDetailPage } from "./pages/admin/quiz-detail.js";

const path = window.location.pathname;

if (path.endsWith("/pages/login.html")) {
  initLoginPage();
}

if (path.endsWith("/pages/signup.html")) {
  initSignupPage();
}

if (path.endsWith("/pages/category.html")) {
  initCategoryPage();
}
```

### 왜 좋은가요?

- `main.js`는 "어느 페이지를 시작할지"만 관리합니다.
- 로그인 로직이 회원가입 로직과 섞이지 않습니다.
- 각 파일이 자기 역할만 맡아서 응집도가 높아집니다.

## 5. 페이지별 JS는 "한 페이지, 한 책임" 원칙으로 작성하세요

### 좋지 않은 방식

- 모든 이벤트를 `main.js` 하나에 몰아넣기
- HTML 안에 `onclick` 쓰기
- `fetch`와 화면 렌더링을 한 함수에 전부 몰아넣기

### 좋은 방식

- `login.js`: 로그인 폼 이벤트만 담당
- `signup.js`: 회원가입 폼 이벤트만 담당
- `category.js`: 카테고리 목록 렌더링과 클릭 이벤트만 담당
- `quiz-list.js`: 관리자 문제 목록 렌더링만 담당

예시:

```js
export function initLoginPage() {
  const form = document.querySelector("#loginForm");

  if (!form) {
    return;
  }

  form.addEventListener("submit", handleSubmit);
}

async function handleSubmit(event) {
  event.preventDefault();
  // 입력값 읽기
  // 검증
  // API 호출
  // 성공/실패 처리
}
```

이 구조는 초보자에게 특히 좋습니다. 이유는 "어디를 고쳐야 할지"가 바로 보이기 때문입니다.

## 6. API 호출도 화면 코드와 분리하세요

지금 `src/api.js`가 비어 있는데, 여기서 중요한 것은 "API 호출을 모아둘 자리"를 제대로 쓰는 것입니다.

예시:

```js
const API_BASE_URL = "https://example.com";

export async function login(payload) {
  const response = await fetch(`${API_BASE_URL}/auth/login`, {
    method: "POST",
    headers: {
      "Content-Type": "application/json",
    },
    body: JSON.stringify(payload),
  });

  if (!response.ok) {
    throw new Error("로그인에 실패했습니다.");
  }

  return response.json();
}
```

그리고 페이지 파일에서는 이렇게 씁니다.

```js
import { login } from "../api.js";

export function initLoginPage() {
  const form = document.querySelector("#loginForm");

  if (!form) {
    return;
  }

  form.addEventListener("submit", async (event) => {
    event.preventDefault();

    const email = document.querySelector("#email").value.trim();
    const password = document.querySelector("#password").value;

    const data = await login({ email, password });
    console.log(data);
  });
}
```

### 왜 분리하나요?

- API 주소가 바뀌면 `api.js`만 보면 됩니다.
- 화면 수정은 페이지 파일만 보면 됩니다.
- 결합도가 낮아져서 수정 범위가 줄어듭니다.

## 7. HTML 기본 속성을 먼저 챙기면 JavaScript가 줄어듭니다

브라우저는 원래 폼을 많이 도와줍니다. 그 기능을 먼저 써야 합니다.

예시:

```html
<label for="email" class="block text-sm">이메일</label>
<input
  id="email"
  name="email"
  type="email"
  required
  autocomplete="email"
  class="w-full rounded-lg border border-black/10 px-4 py-3 focus:outline-none focus:ring-2 focus:ring-black/20"
  placeholder="이메일을 입력하세요"
/>
```

회원가입 페이지에서는 아래도 고려하세요.

- `nickname`: `name`, `required`, `minlength`, `maxlength`
- `password`: `name`, `required`, `autocomplete="new-password"`
- 약관 체크박스: `required`
- 관리자 문제 등록 폼: `title`, `category`, `difficulty`, `question`, `explanation`, `option1~4`에 `name`과 `required`

### 왜 중요한가요?

- 브라우저가 기본 검증을 해줍니다.
- JavaScript 검증 코드가 줄어듭니다.
- 접근성과 사용성이 좋아집니다.

## 8. 인라인 이벤트 대신 `addEventListener`를 사용하세요

지금은 이런 코드가 있습니다.

```html
<button onclick="filterCategory(this, 'HTML')">HTML</button>
```

이 방식은 HTML과 JavaScript를 섞습니다. 대신 아래처럼 분리하세요.

```html
<button class="filter-btn" data-category="HTML">HTML</button>
```

```js
document.querySelectorAll(".filter-btn").forEach((button) => {
  button.addEventListener("click", () => {
    const category = button.dataset.category;
    filterCategory(category);
  });
});
```

### 장점

- HTML은 구조에 집중합니다.
- JavaScript는 동작에 집중합니다.
- 테스트하고 수정하기 쉬워집니다.

## 9. Tailwind는 유지하되, 반복 마크업은 줄이세요

여기서 중요한 점은 "반복을 줄인다"가 "CSS 파일을 많이 만든다"는 뜻이 아니라는 것입니다.

이 프로젝트는 계속 Tailwind 중심으로 가는 편이 좋습니다. 대신 아래처럼 정리하세요.

### 방법 1. 반복 카드 데이터를 배열로 만들기

```js
const categories = [
  { key: "html", label: "HTML", description: "HTML 기초를 학습합니다." },
  { key: "css", label: "CSS", description: "CSS 기초를 학습합니다." },
  { key: "js", label: "JavaScript", description: "JavaScript 기초를 학습합니다." },
];
```

### 방법 2. 템플릿 함수로 렌더링하기

```js
function createCategoryCard(category) {
  return `
    <button
      type="button"
      data-category="${category.key}"
      class="w-full rounded-lg border border-black/10 p-6 text-left transition-colors hover:bg-black/5"
    >
      <h3 class="font-medium">${category.label}</h3>
      <p class="mt-1 text-sm text-black/60">${category.description}</p>
    </button>
  `;
}
```

### 방법 3. 정말 자주 쓰는 클래스만 상수로 분리하기

```js
const primaryButtonClass =
  "w-full rounded-lg bg-black py-4 text-white transition-colors hover:bg-black/90 disabled:cursor-not-allowed disabled:opacity-50";
```

이 방법은 CSS를 거의 늘리지 않으면서도 중복을 줄여줍니다.

## 10. 추천 작업 순서

초보 개발자라면 아래 순서가 가장 안전합니다.

1. 파일명과 링크를 실제 구조에 맞게 통일합니다.
2. `vite.config.js`를 고쳐서 빌드가 되게 만듭니다.
3. `npm run build`로 먼저 배포 가능 상태를 확인합니다.
4. `main.js`에서 페이지별 초기화 구조를 만듭니다.
5. 로그인 페이지 하나만 먼저 완성합니다.
6. 회원가입 페이지를 같은 방식으로 완성합니다.
7. 카테고리 선택과 히스토리를 데이터 기반 렌더링으로 정리합니다.
8. 관리자 목록, 등록, 상세, 수정 페이지를 같은 패턴으로 확장합니다.
9. 마지막으로 README를 실제 구조에 맞게 업데이트합니다.
