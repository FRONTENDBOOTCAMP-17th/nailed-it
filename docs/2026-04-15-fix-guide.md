# 2026-04-15 코드 리뷰 해결 가이드

## 이 문서는 어떻게 읽으면 좋나요?

이 문서는 위 코드 리뷰에서 나온 문제를 "초보 개발자가 실제로 어떻게 고치면 되는지"를 설명하는 문서입니다.

- 목표 1: 응집도는 높이고 결합도는 낮추기
- 목표 2: CSS를 최소화하고 Tailwind CSS 중심으로 작성하기
- 목표 3: HTML, JavaScript, API 연결의 역할을 나누기

---

## 1. 먼저 구조부터 바로잡아야 합니다

현재 가장 큰 문제는 "화면 파일은 여러 개 있는데, 구조가 그 파일들을 제대로 관리하지 못한다"는 점입니다.

추천 구조는 아래처럼 잡는 것이 좋습니다.

```text
NAILED-IT/
├── index.html
├── login.html
├── signup.html
├── categories.html
├── admin-login.html
├── src/
│   ├── main.js
│   ├── style.css
│   ├── js/
│   │   ├── api/
│   │   │   ├── auth.js
│   │   │   ├── categories.js
│   │   │   └── sessions.js
│   │   ├── pages/
│   │   │   ├── login-page.js
│   │   │   ├── signup-page.js
│   │   │   ├── admin-login-page.js
│   │   │   └── categories-page.js
│   │   ├── utils/
│   │   │   ├── storage.js
│   │   │   ├── validation.js
│   │   │   └── dom.js
│   │   └── constants/
│   │       └── routes.js
└── docs/
```

### 왜 이렇게 나누나요?

- `api/`: 서버와 통신하는 코드만 모읍니다.
- `pages/`: 각 페이지에서 필요한 이벤트 연결과 화면 로직을 모읍니다.
- `utils/`: 여러 파일에서 같이 쓰는 작은 함수들을 모읍니다.
- `constants/`: 경로나 고정값을 한곳에서 관리합니다.

이렇게 나누면 각 파일이 맡는 일이 분명해집니다. 이것이 응집도를 높이는 첫걸음입니다.

## 2. `main.js`는 "진입점" 역할만 하게 만드세요

지금 `main.js`는 CSS만 불러오고 있습니다. 앞으로는 "현재 페이지가 무엇인지 보고, 해당 페이지 스크립트를 실행하는 파일"로 쓰는 것이 좋습니다.

예를 들면 아래 흐름입니다.

```js
import "./style.css";
import { initLoginPage } from "./js/pages/login-page.js";
import { initSignupPage } from "./js/pages/signup-page.js";
import { initAdminLoginPage } from "./js/pages/admin-login-page.js";
import { initCategoriesPage } from "./js/pages/categories-page.js";

const path = window.location.pathname;

if (path.endsWith("/login.html")) {
  initLoginPage();
}

if (path.endsWith("/signup.html")) {
  initSignupPage();
}

if (path.endsWith("/admin-login.html")) {
  initAdminLoginPage();
}

if (path.endsWith("/categories.html")) {
  initCategoriesPage();
}
```

### 이 방식의 장점

- `main.js`는 "어떤 페이지를 시작할지"만 결정합니다.
- 로그인 페이지의 상세 로직이 `main.js`에 섞이지 않습니다.
- 페이지가 늘어나도 파일 역할이 깔끔하게 유지됩니다.

## 3. 폼마다 페이지 전용 파일을 만드세요

### 좋지 않은 방식

- 로그인 코드, 회원가입 코드, 관리자 로그인 코드를 모두 한 파일에 몰아넣기
- HTML 안에 `onclick`을 직접 쓰기

### 좋은 방식

- `login-page.js`는 로그인 폼만 담당
- `signup-page.js`는 회원가입 폼만 담당
- `admin-login-page.js`는 관리자 로그인 폼만 담당

예를 들어 로그인 페이지는 아래 순서로 동작하면 됩니다.

1. `#loginForm`을 찾습니다.
2. 없으면 아무 일도 하지 않고 종료합니다.
3. `submit` 이벤트를 연결합니다.
4. 입력값을 읽습니다.
5. 기본 검증을 합니다.
6. `auth.js`의 로그인 API 함수를 호출합니다.
7. 성공하면 사용자 역할에 따라 페이지를 이동합니다.
8. 실패하면 `#errorMessage`에 메시지를 보여줍니다.

이렇게 하면 페이지 파일 하나가 "한 화면의 동작"만 책임지게 됩니다. 이것이 응집도를 높이는 좋은 예시입니다.

## 4. API 호출 코드는 화면 파일에서 분리하세요

초보 개발자들이 자주 하는 실수는 `fetch()`를 HTML 이벤트 코드 안에 바로 길게 적는 것입니다.

예를 들어 아래처럼 섞이면 관리가 어렵습니다.

```js
form.addEventListener("submit", async (event) => {
  event.preventDefault();
  const response = await fetch("...");
  const data = await response.json();
  // 화면 처리
});
```

처음에는 빨라 보이지만, 나중에는 로그인 실패 처리, 토큰 저장, 에러 메시지 정리 때문에 코드가 금방 복잡해집니다.

### 추천 방식

`src/js/api/auth.js`

```js
export async function login(payload) {
  const response = await fetch("/auth/login", {
    method: "POST",
    headers: {
      "Content-Type": "application/json",
    },
    body: JSON.stringify(payload),
  });

  if (!response.ok) {
    throw new Error("로그인에 실패했습니다.");
  }

  return response.json();
}
```

`src/js/pages/login-page.js`

```js
import { login } from "../api/auth.js";

export function initLoginPage() {
  const form = document.querySelector("#loginForm");

  if (!form) {
    return;
  }

  form.addEventListener("submit", async (event) => {
    event.preventDefault();
    // 입력값 읽기
    // 검증하기
    // login 호출하기
    // 성공/실패 처리하기
  });
}
```

### 왜 좋은가요?

- API가 바뀌면 `api/auth.js`만 보면 됩니다.
- 로그인 화면을 수정할 때는 `login-page.js`만 집중해서 보면 됩니다.
- 결합도가 낮아져서 수정 범위가 줄어듭니다.

## 5. Tailwind는 계속 쓰되, "반복 마크업"을 줄이세요

여기서 중요한 점이 있습니다.

반복을 줄인다고 해서 곧바로 CSS 파일을 늘리라는 뜻은 아닙니다.

이 프로젝트에서는 CSS를 최소화하고 Tailwind를 계속 유지하는 편이 좋습니다. 대신 아래 방법을 추천합니다.

### 방법 1. JavaScript 데이터 배열로 렌더링하기

현재 `categories.html`에는 HTML, CSS, JavaScript 카드가 거의 같은 구조로 반복됩니다.

이런 경우에는 데이터 배열을 만들고 `map()`으로 렌더링하는 방식이 좋습니다.

```js
const categories = [
  { key: "html", label: "HTML", description: "HTML 기초를 학습합니다." },
  { key: "css", label: "CSS", description: "CSS 기초를 학습합니다." },
  { key: "js", label: "JavaScript", description: "JavaScript 기초를 학습합니다." },
];
```

이후 공통 카드 마크업 함수를 만들어서 반복 렌더링하면 됩니다.

### 방법 2. 자주 쓰는 클래스 묶음을 상수로 분리하기

예를 들면 버튼 클래스가 자주 반복된다면 아래처럼 문자열 상수로 둘 수 있습니다.

```js
const primaryButtonClass =
  "w-full rounded-lg bg-black py-4 text-white transition-colors hover:bg-black/90 disabled:cursor-not-allowed disabled:opacity-50";
```

이 방법은 CSS를 추가하지 않으면서도 중복을 줄일 수 있습니다.

### 방법 3. 정말 필요한 경우에만 작은 커스텀 클래스 사용하기

정말 같은 Tailwind 조합이 너무 많이 반복되면 `style.css`에 아주 작은 유틸 클래스를 둘 수는 있습니다.

하지만 이때도 원칙은 같습니다.

- Tailwind로 해결 가능한 것은 Tailwind로 해결
- 커스텀 CSS는 최소한으로만 사용

## 6. 기본 HTML 속성부터 챙기세요

JavaScript를 많이 쓰기 전에 HTML 기본기를 먼저 챙기면 전체 코드가 쉬워집니다.

### 로그인/회원가입 폼에서 바로 추가하면 좋은 것

- `label`에 `for`
- `input`에 `id`
- `input`에 `name`
- `required`
- `autocomplete`
- 필요하면 `minlength`, `maxlength`, `pattern`

예시:

```html
<label for="email" class="block text-sm">이메일</label>
<input
  id="email"
  name="email"
  type="email"
  required
  autocomplete="email"
  class="w-full rounded-lg border border-black/10 px-4 py-3 focus:outline-none focus:ring-2 focus:ring-black/20"
/>
```

### 왜 중요한가요?

- 브라우저가 자동으로 기본 검증을 도와줍니다.
- 모바일 키보드가 입력 타입에 맞게 바뀝니다.
- 접근성이 좋아집니다.
- JavaScript가 덜 복잡해집니다.

## 7. 인라인 이벤트는 없애고 `addEventListener`를 쓰세요

현재 로그인 페이지에는 아래 코드가 있습니다.

```html
<button type="button" onclick="alert('비밀번호 찾기 기능은 추후 구현 예정입니다.')">
```

처음에는 편하지만, 이 방식은 금방 관리가 어려워집니다.

더 좋은 방식은 아래 흐름입니다.

1. HTML에는 버튼만 둡니다.
2. JavaScript에서 버튼을 선택합니다.
3. `addEventListener("click", ...)`로 이벤트를 연결합니다.

이렇게 하면 HTML은 구조만 맡고, JavaScript는 동작만 맡습니다.

현재 README에는 여러 페이지가 있는 것처럼 적혀 있지만, 실제 파일은 아직 없습니다.

이런 상태가 계속되면 팀원들이 헷갈립니다.

- 이미 만든 파일만 README에 "현재 구조"로 적기
- 아직 만들 예정인 것은 "예정 구조"로 따로 적기

문서와 코드를 같이 관리하는 습관은 협업에서 매우 중요합니다.

## 9. 추천 작업 순서

초보 개발자라면 아래 순서로 작업하는 것이 가장 안전합니다.

### 1단계. 배포 구조 정리

- Vite에서 여러 HTML 페이지가 빌드되도록 설정합니다.
- `dist/`에 어떤 HTML이 생성되는지 꼭 확인합니다.

### 2단계. 페이지별 JavaScript 파일 나누기

- 로그인
- 회원가입
- 관리자 로그인
- 카테고리 선택

각 페이지마다 init 함수를 하나씩 만듭니다.

### 3단계. API 파일 분리

- `auth.js`
- `categories.js`
- `sessions.js`

서버 통신 코드는 페이지 파일에서 분리합니다.

### 4단계. 기본 HTML 품질 올리기

- `required`
- `name`
- `autocomplete`
- `label for`

### 5단계. 반복 UI 줄이기

- 카테고리 카드 데이터화
- 공통 버튼 클래스 상수화
- 필요한 경우에만 최소한의 CSS 사용

### 6단계. 없는 링크 정리

- `history.html` 등 미구현 링크 제거 또는 실제 구현

## 10. 응집도와 결합도를 이 프로젝트에 맞게 쉽게 설명하면

### 응집도가 높다

- 로그인 파일은 로그인 일만 한다
- 회원가입 파일은 회원가입 일만 한다
- API 파일은 서버 통신만 한다

즉, 한 파일이 "한 가지 일"에 집중하면 응집도가 높다고 볼 수 있습니다.

### 결합도가 낮다

- 로그인 화면을 수정해도 회원가입 코드가 거의 안 바뀐다
- 카테고리 기능을 바꿔도 인증 함수는 거의 안 건드린다
- 버튼 스타일을 정리해도 모든 페이지를 복붙 수정하지 않는다

즉, 한 부분을 바꿨을 때 다른 부분이 덜 흔들리면 결합도가 낮다고 볼 수 있습니다.

## 마지막 정리

지금 단계에서 가장 중요한 것은 "화면을 더 많이 만드는 것"이 아닙니다.

더 중요한 것은 아래 3가지입니다.

1. 페이지가 실제로 동작하도록 구조를 나누기
2. API 호출, 화면 로직, 공통 유틸의 역할을 분리하기
3. Tailwind 중심을 유지하면서 반복을 줄이기

이 3가지를 먼저 잡아두면 이후에 퀴즈 화면, 결과 화면, 히스토리 화면을 추가할 때 훨씬 덜 힘들어집니다.
